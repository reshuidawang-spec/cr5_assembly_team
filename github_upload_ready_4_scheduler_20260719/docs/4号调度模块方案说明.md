# 4号调度模块方案说明

## 一、装配元件如何细化

原来的调度主要是按“大工序”进行，例如箱体上料、PCB安装、控制模块安装、端子排安装等。为了让调度过程更接近真实低压配电柜装配，我将装配对象进一步细化为“主元件 + 细节特征”。

当前细化后的装配对象包括：箱体壳、PCB板、PCB定位孔、PCB板载元件、控制模块本体、控制模块标签、端子排本体、端子排接线槽、端子排主螺钉头等。

其中，箱体壳、PCB板、控制模块本体、端子排本体属于主要安装元件；PCB定位孔、板载元件、控制模块标签、端子排接线槽、主螺钉头属于细节特征，主要用于表达定位、检测和锁付对象。这样调度不再只是简单安排机械臂执行任务，而是能体现“先装什么、后装什么、每一步具体操作什么”。

## 二、工序流程是什么样的

根据五机械臂低压配电柜装配场景，整体工序流程为：

```text
R1 箱体上料
 -> R2 PCB安装
 -> R3 控制模块安装
 -> R1 端子排安装
 -> R3 转移完整装配体到检测区
 -> CAMERA 固定相机检测
 -> R4 螺钉锁付
 -> R5 合格品/缺陷品分拣
```

每台设备的分工为：

```text
R1：箱体上料、端子排安装
R2：PCB安装
R3：控制模块安装、装配体转移
CAMERA：固定相机检测
R4：端子螺钉锁付
R5：检测后分拣
```

同时，我使用层次化拓扑排序确定装配顺序。也就是说，如果某个元件必须依赖前一个元件安装完成，它就会排在后面。例如，没有箱体就不能安装PCB，没有PCB就不能安装控制模块，没有端子排就不能进行后续锁付。

## 三、我的综合调度方案流程

我的方案是“综合优化调度”。它不是只按照一个规则排序，而是把订单紧急程度、交期、等待时间、后续工序压力、瓶颈资源和共享区域冲突一起考虑。

整体流程如下：

```text
读取订单
 -> 获取产品装配元件和工序信息
 -> 根据依赖关系生成装配任务
 -> 使用层次化拓扑排序确定安装顺序
 -> 将工序进一步拆分为工步
 -> 找出当前可以执行的候选任务
 -> 对候选任务进行综合评分
 -> 选择评分较高的任务优先执行
 -> 分配给空闲的 R1-R5 或 CAMERA
 -> 执行过程中记录开始时间、结束时间、等待时间
 -> 输出时间轴仿真和装配线平衡结果
 -> 遇到故障时暂停相关资源，维修后重新调度
```

### 1. 候选任务如何产生

不是所有任务都能立刻执行，只有满足以下条件的任务才会进入候选队列：

```text
前置工序已经完成
订单已经到达
对应机械臂或相机空闲
所需区域没有被其他任务占用
```

例如，PCB安装必须等箱体上料完成；R4锁付必须等相机检测完成；R5分拣必须等检测和锁付完成后才能进行。

### 2. 综合评分怎么评

每个候选任务都会计算一个综合分数，分数越高越优先执行。评分主要由以下几部分组成：

```text
综合评分 =
订单优先级得分
+ 交期紧急程度得分
+ 等待时间得分
+ 剩余关键路径得分
- 瓶颈资源惩罚
- 共享区域冲突惩罚
```

各部分含义如下：

```text
订单优先级得分：
优先级越高，得分越高。priority >= 5 的订单会被看作急单。

交期紧急程度得分：
如果订单距离交期越近，或者已经存在延误风险，得分越高。

等待时间得分：
任务等待越久，得分会逐渐提高，避免某些普通任务一直被压在后面。

剩余关键路径得分：
如果某个任务后面还跟着较长的后续工序，说明它对整体完成时间影响较大，会适当提高优先级。

瓶颈资源惩罚：
当前装配线分析显示 R3 负载较重，所以普通订单如果继续占用 R3，会受到轻微惩罚，避免瓶颈继续拥堵。

共享区域冲突惩罚：
检测平台是 CAMERA、R4、R5 都会涉及的共享区域，如果任务容易造成检测、锁付、分拣之间的拥堵，会受到一定惩罚。
```

其中，急单不会被瓶颈惩罚压下去。也就是说，如果是急单，即使它需要 R3 或检测平台，也仍然会优先考虑。

### 3. 为什么这样评分

这样评分的目的不是单纯追求“总完工时间最短”，而是更符合实际生产需求：

```text
急单要尽快完成
快到交期的订单要优先处理
等待太久的任务不能一直被忽略
关键路径上的任务不能拖延
瓶颈机械臂不能被普通订单持续挤占
共享检测平台不能发生明显拥堵
```

因此，我的方案更偏向柔性调度，而不是简单排序。

### 4. 装配线平衡如何参与

在仿真结束后，我统计每个资源的工作负载，包括 R1、R2、R3、R4、R5 和 CAMERA。通过比较各资源的工作时间，可以找出瓶颈资源。

当前仿真中，R3 的负载较高，因为它既要安装控制模块，又要负责将完整装配体转移到检测区。因此，我在评分中加入了 R3 瓶颈惩罚，让普通订单适当避开 R3 的高峰占用。

装配线平衡不是强行把任务平均分给所有机械臂，因为每台机械臂的功能不同，例如 R4 只能做锁付，CAMERA 只能检测。这里的平衡主要是通过调度顺序减少拥堵和等待。

### 5. 故障发生时如何处理

如果生产过程中某台机械臂或相机发生故障，调度器会进行如下处理：

```text
故障资源暂停接收新任务
如果故障发生时正在执行任务，则该任务中断
释放该任务占用的区域
等待维修完成
维修后该任务重新进入候选队列
调度器重新评分并分配任务
```

例如，如果 R4 在锁付过程中故障，那么锁付任务会暂停，检测平台相关区域会释放或等待，R4维修完成后再重新调度锁付任务。

### 6. 输出结果

我的方案会输出以下结果：

```text
装配元件顺序表
工步级时间轴
装配线平衡结果
四种方案对比表
机械臂/相机故障修复仿真结果
可视化 HTML 页面
```

这些结果可以用来说明调度过程是否合理、急单是否被优先处理、哪个机械臂是瓶颈，以及故障后系统是否还能恢复生产。

## 四、三个对比方案流程

为了说明综合优化方案的特点，我设置了三个对比方案。

### 方案A：最基本串行

流程为：

```text
订单进入
 -> 按顺序执行第一个任务
 -> 完成后再执行下一个任务
 -> 一次只处理一个任务
```

这个方案最简单，但不能发挥五台机械臂并行协作的优势。

### 方案B：基础并行 FIFO

流程为：

```text
订单进入
 -> 多台机械臂可以同时工作
 -> 任务按先来先服务顺序执行
 -> 只处理区域互斥，不做综合评分
```

这个方案利用了多机械臂并行能力，但不区分急单，也不考虑交期和瓶颈。

### 方案C：单一关键路径优先

流程为：

```text
订单进入
 -> 计算任务剩余工作量或关键路径
 -> 优先执行看起来更关键的任务
 -> 不考虑急单、交期、等待时间和瓶颈资源
```

这个方案比普通 FIFO 更“算法化”，但它只看一个因素。如果只看关键路径，可能会忽略急单，也可能造成某些资源拥堵。

## 五、总结

本次调度工作从原来的工序级调度，进一步扩展为“装配元件 + 工步 + 资源 + 时间 + 故障恢复”的综合调度模型。整体上，我的方案更关注实际生产中的柔性调度需求，包括急单处理、资源冲突、瓶颈机械臂和故障恢复。
